# 5. Selective Disclosure

## 5.1 Approach

UC 10 uses **SD-JWT VC** as the credential format for all attestations. Selective disclosure is achieved at the protocol level: each Relying Party requests only the attributes it needs, and the EUDIW constructs a Verifiable Presentation containing only the requested disclosures. The user sees exactly which attributes are being shared before consenting (see Section 1.2).

---

## 5.2 Per-RP Attribute Disclosure Matrix

The following matrix defines which attributes are disclosed to each Relying Party. This is the primary data minimisation artefact for UC 10.

| Relying Party | Credential | Disclosed Attributes | Undisclosed Attributes |
|---------------|------------|----------------------|------------------------|
| Booking Platform (Phase 2) | PID | `family_name`, `given_name`, `birth_date`, `nationality` | All other PID attributes (`address`, `document_number`, `place_of_birth`, etc.) |
| Hotel Check-in (Phase 3) | PID | `family_name`, `given_name`, `birth_date`, `document_number`, `nationality` | `address`, `place_of_birth`, and other optional PID attributes |
| Hotel Check-in (Phase 3) | Booking Confirmation EAA | `booking_id`, `hotel_id`, `check_in_date`, `check_out_date`, `guest_family_name`, `guest_given_name` | `room_type`, `number_of_guests` (not needed for identity matching) |
| Door Reader (Phase 4) | Digital Room Key EAA | All attributes (`room_number`, `valid_from`, `valid_until`, `lock_mac_key`, `lock_protocol`) | None — all attributes required for door unlock |
| Authorities (Phase 5) | PID | `family_name`, `given_name`, `birth_date`, `nationality`, `document_number`, `place_of_birth` | `address` and any other non-required attributes |

---

## 5.3 Selective Disclosure for Door Key EAA

As noted in the HLR, selective disclosure is not required for the Digital Room Key EAA. All attributes in this credential are necessary for the door unlock operation, and the credential was designed with data minimisation at issuance: it contains only the minimum data needed for room access. This is consistent with a **"privacy by design"** approach where minimisation is applied at credential design time rather than at presentation time.

---

## 5.4 Derived Attributes

During the identity verification for door key issuance (Phase 3–4 transition), derived attributes may be verified. For example, the hotel system may verify that the guest "is the booking holder" by matching PID name attributes against booking records. However, no derived-attribute claims (e.g., `is_over_18`, `is_eligible`) are issued or disclosed as standalone attestations in UC 10.
