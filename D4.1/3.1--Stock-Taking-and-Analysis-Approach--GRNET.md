## 3.1 Stock-Taking and Analysis Approach

This section describes the overall approach followed during the T4.2 stock-taking phase: how the set of 11 accepted use cases was assembled from an initially larger pool of candidate ideas, how the partner constellation of each use case was shaped, and how the analysis itself was structured as the specifications were progressively filled in. 

### From initial UC ideas to the final set of 11 accepted use cases

The stock-taking phase began with a bottom-up collection of use case ideas proposed by the partners who would eventually take the UC lead role. Each UC lead was free to put forward a use case idea that reflected a real business need within its domain (airline operations, travel distribution, student mobility, hotel check-in, stadium access, cross-border train travel, and so on). Already at this very early stage, the UC leads were responsible for bringing on board the other partners that each use case would require in order to be feasible: credential issuers (such as airlines, universities, student card providers), relying parties (such as hotels, airports, transport operators, event venues), wallet providers from participating Member States, and any other role without which the use case could not be piloted end to end. This partner-finding activity is a defining trait of APTITUDE as a Member-State-driven initiative: it ensures that every accepted use case is backed by a concrete consortium able to pilot it.

The initial harvest produced more than 30 use cases. A screening and consolidation activity was then carried out by the WP4 lead (GRNET), in close dialogue with the UC leads. Two main simplifications were applied:

1. **Merging of overlapping use cases.** Where two or more candidate use cases described essentially the same user journey, the same credential types, and the same actors, they were merged into a single use case with a combined partner constellation. This avoided artificial fragmentation of the WP4 pilot portfolio and reduced the cognitive overload and focus distraction inherent in a larger number of use cases.
2. **Grouping under SEDIT-X.** A number of candidate use cases described individual touchpoints along a single end-to-end traveller journey (smart airport in Thessaloniki, urban mobility in Athens, ferry transport, hotel check-in, academic access). These were united into a single overarching use case led by the University of the Aegean, called SEDIT-X (Seamless Digital Traveller Experience), structured as a multi-episode use case. Each episode retains its own specification section, but the five episodes are presented as a coherent whole in order to highlight the multi-modal nature of the journey.

The outcome of this consolidation activity is the final set of **11 accepted use cases** that is presented throughout the rest of this deliverable. The list, with UC numbers, titles, and leads, is given in Chapter 1 (Executive Summary) and expanded upon in Chapter 8.

### A two-stage specification process

Once the 11 accepted use cases were fixed, the specification effort focused on producing detailed specification sheet from the original lightweight sketches.

**Stage 1: Lightweight spreadsheet.** We maintained a simple shared spreadsheet in which each UC was represented by a single row. The columns captured only the essential metadata needed to understand the UC at a glance: the UC title, the UC lead, the Member States involved, a high-level description of the scenario, the list of actors required (credential issuers, relying parties, wallet providers, end users), and a preliminary mapping to the three WP4 scenarios. The goal at this stage was not to describe the UC in technical depth but rather to give the consortium a common view of the UC portfolio, to surface dependencies between use cases, and to identify the gaps in partner coverage that needed to be closed. This spreadsheet also supported the coordination discussions with WP3 (on Digital Travel Credentials) and with WP6 (on payment steps), making these footprints visible.

**Stage 2: Standardised UC specification template.** Once the UC portfolio was stable, GRNET designed a standardised specification template to be filled in by every UC lead. The template expands on the spreadsheet metadata with a structured set of fields covering, among others, the UC summary, the user stories, the actor definitions, the context and preconditions, the credentials involved, the step-by-step user journey, the technical flow with accompanying diagrams, and the unhappy paths. For each field, the template provides a short description that explains what is expected and, where helpful, an exemplary value drawn from a representative use case, so that UC leads from very different domains (air travel, hospitality, student mobility) can fill the template consistently. The rationale behind the template, together with the full list of fields and their definitions, is presented in dedicated detail in Section 3.4. All 11 UC leads filled the template for their respective use case, and the resulting specification sheets are included in full in Annex A. The narrative of each use case, drawn from those sheets, is presented in Chapter 8.

### Structure of the analysis behind the specifications

The analysis that underlies the specifications is structured along three intentionally ordered layers, reflected in the organisation of this deliverable:

1. **Business requirements.** The starting point for each use case is the business-level need that motivates it: what is the current process, where does it fall short today, and what should the EUDIW-enabled process achieve in terms of user experience, efficiency, or compliance? These business requirements emerge from the baseline analysis of existing processes and from the stakeholder engagement carried out by the UC leads. The baseline findings that support this layer are presented in Chapter 4.
2. **Legal and regulatory analysis.** Every use case is then examined against the applicable legal framework, starting from eIDAS 2.0 and its Implementing Acts, and extending to sector-specific regulations (for example, national hotel guest registration laws, passenger rights regulations, rules governing the cross-border recognition of student status). This layer ensures that the specifications are not only technically feasible but also legally sound. The consolidated legal analysis is presented in Chapter 5, with sector-specific subsections.
3. **Functional and non-functional interoperability specifications.** Finally, each UC is specified in terms of functional behaviour (what the wallet, the credential issuers, and the relying parties must do), and in terms of non-functional properties that the pilot implementations must exhibit (for example, response time at airport gates, offline capability for ferry boarding, accessibility for users with reduced mobility). Cross-border interoperability cuts across both dimensions and is treated as a first-class requirement. The common technical foundations that underpin this layer are described in Chapter 6, and the UC-specific specifications appear in Chapter 8 and in Annex A.

This ordering ensures that every technical decision documented in the deliverable can be traced back to a concrete business need and checked against the applicable legal constraints. It also supports the traceability that will be needed later in the project, when D4.2 translates these specifications into implementation guidance, and when D4.3 evaluates whether the piloted implementations actually meet the original objectives.

### Iterative refinement

The stock-taking process was an iterative activity. Between the initial UC ideas and the specification sheets presented in Annex A, multiple iterative refinement cycles took place between the UC leads and the WP4 coordination team. These cycles included bilateral calls with each UC lead, dedicated workshops (some scenario-focused, some cross-cutting), and written reviews of draft specifications. Each cycle resulted in a more refined set of specifications. The stakeholders consulted during these cycles, and the tools that supported them, are documented in Section 3.2 and Section 3.3 respectively, with a dedicated subsection per UC so that the evidence base for each individual use case can be traced.

### Diagrammatic Conventions and the Use of Mermaid

Several fields of the UC specification template call for diagrams (user journeys, technical flows, unhappy paths) to help visualize business or technical flows. To keep these diagrams consistent across the 11 use cases, and to ensure that they can be authored, reviewed, and rendered without specialised tooling, WP4 has adopted **Mermaid** as the single diagramming language used in D4.1.

Mermaid is an open-source, text-based diagramming language. Instead of drawing boxes and arrows in a graphical editor, the author writes a short declarative description, which a renderer turns into a visual diagram. 
Mermaid was chosen for several reasons:

* **Diagrams as text.** Mermaid sources live in the same Markdown file as the surrounding text, rather than as separate binary artifacts (Visio, PowerPoint, or screenshots). Changes appear as readable line-by-line diffs in pull requests, which matters in a consortium where 11 UC leads contribute concurrently.
* **Native GitHub rendering.** Since 2022, GitHub renders Mermaid code blocks automatically inside Markdown files, pull requests, and issues. No plugins or build steps are needed for reviewers to see the diagram directly in the browser.
* **Low barrier to entry.** Authors unfamiliar with the syntax can produce a readable flowchart in minutes, and labels remain searchable and accessible because they stay in the text.
