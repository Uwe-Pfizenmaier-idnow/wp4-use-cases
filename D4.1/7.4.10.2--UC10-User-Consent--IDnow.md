# 1. User Consent

## 1.1 Consent Points in the UC Flow

UC 10 requires explicit user consent at four distinct points in the journey. Each consent prompt is triggered by the EUDIW and requires user authentication (biometrics or PIN) before data is shared.

| Phase | Consent Trigger | Data Shared | Relying Party | Legal Basis |
|-------|----------------|-------------|---------------|-------------|
| **Phase 2 — Booking** | Booking Platform requests PID attributes via OID4VP | PID subset (name, DOB, nationality) | Booking Platform (Participant 1) | GDPR Art. 6(1)(a) — consent; eIDAS 2.0 Art. 5a |
| **Phase 3 — Check-in** | Hotel requests PID + Booking EAA via OID4VP | PID subset + Booking Confirmation EAA | Hotel (Participant 2) | GDPR Art. 6(1)(a) + Art. 6(1)(c) — legal obligation (guest registration) |
| **Phase 4 — Room key** | Door reader requests Room Key EAA via OID4VP | Digital Room Key EAA | Hotel door lock system | GDPR Art. 6(1)(b) — contractual necessity |
| **Phase 5 — Authority reporting** | Hotel requests guest data sharing with authorities | PID subset (minimised) | Border / Local Registry authorities | GDPR Art. 6(1)(c) — legal obligation; EU Reg. 2025/12 |

## 1.2 Consent Mechanism

At each consent point, the EUDIW displays the following information to the user before requesting authentication:
- Relying Party identity: The name and registration details of the requesting party (e.g., hotel name, booking platform name). For Phase 4 (door unlock), the RP identifier is the room number.
- Requested attributes: The specific attributes the RP has requested, displayed as a human-readable list (e.g., "Family name, Given name, Date of birth, Nationality").
- Purpose statement: A brief explanation of why the data is being requested (e.g., "To verify your identity for hotel check-in" or "To share guest registration data with local authorities as required by law").
- Authentication requirement: The user must authenticate via biometrics or wallet PIN before data is released. No silent or automatic sharing occurs.

## 1.3 Consent Granularity

Consent in UC 10 operates on a per-presentation basis: the user either approves or declines the entire set of attributes requested by the RP. Attribute-level selective consent (approving some attributes while declining others within a single presentation) is not supported in the current flow. However, data minimisation is achieved through selective disclosure at the protocol level (see Section 5), ensuring that only the minimum required attributes are requested by each RP.

## 1.4 Consent Withdrawal

Once a Verifiable Presentation has been shared, the data cannot be "unshared" from the RP. However:
- The user may decline consent at any subsequent presentation point without affecting earlier presentations.
- For Phase 5 (authority reporting), if the user declines consent, Unhappy Path 3 is triggered: the hotel falls back to the regulatory fallback process (manual guest registration).
- No ongoing consent relationship exists between the user and the RP — each presentation is a one-time, point-in-time data share.

## 1.5 Consent Logging

The EUDIW maintains an internal transaction log recording each presentation event (timestamp, RP identifier, attributes shared). This log is accessible only to the user. The Relying Party (hotel, booking platform) retains evidence of the presentation for the duration required by applicable regulation:
- Hotel check-in evidence: retained for the duration of the stay plus any legally mandated retention period under national hotel registration law.
- Authority data: retained per the requirements of EU Regulation 2025/12 and national transpositions.
