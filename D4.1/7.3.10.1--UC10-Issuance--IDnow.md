# 2. Issuance

## 2.1 Issuance Overview

Three Electronic Attestations of Attributes (EAAs) are issued within UC 10, all by IDnow acting as QTSP. All EAAs are issued in SD-JWT VC format via OIDC4VCI.

| EAA | Issuer | Trigger | OIDC4VCI Flow | Bound To | Validity |
|-----|--------|---------|---------------|----------|----------|
| **Booking Confirmation EAA** | IDnow (QTSP) | Booking Platform requests issuance after successful booking (Phase 2) | Pre-authorised code flow | Wallet instance (device key binding) | Until checkout date |
| **Digital Room Key EAA** | IDnow (QTSP) | Hotel requests issuance after successful check-in (Phase 4) | Pre-authorised code flow | Wallet instance (device key binding) | Check-in to checkout (typically 1–5 days) |
| **Guest Summary EAA** | IDnow (QTSP) | Hotel requests issuance alongside Room Key (Phase 4) | Pre-authorised code flow | Wallet instance (device key binding) | Duration of stay |

## 2.2 OIDC4VCI Flow Detail

UC 10 uses the pre-authorised code flow variant of OIDC4VCI for all three EAAs. This flow is selected because:
- The issuance is always triggered by a trusted party (Booking Platform or Hotel) that has already authenticated the user via PID verification.
- No additional end-user authorisation step at the issuer (IDnow) is needed beyond the PID-based identity verification already completed.
- The pre-authorised code flow reduces round-trips and latency, which is critical for the in-person check-in scenario (Phase 4).
Flow sequence (Phase 4 example — Room Key issuance):
- 1. Hotel system sends an issuance request to the EUDIW, including a pre-authorised code from IDnow.
- 2. EUDIW contacts IDnow's OIDC4VCI credential endpoint with the pre-authorised code.
- 3. IDnow validates the code, confirms the hotel's identity and the guest's PID binding, and issues the SD-JWT VC.
- 4. EUDIW stores the credential and confirms receipt.

## 2.3 Credential Binding

Each EAA is cryptographically bound to the wallet instance via a device-held key. The SD-JWT VC includes a confirmation claim (cnf) referencing the wallet's public key. This ensures that the credential can only be presented by the wallet to which it was issued, preventing credential transfer or replay.

## 2.4 Issuer Authentication

IDnow signs all EAAs using a qualified electronic seal certificate issued under eIDAS 2.0. The certificate chain is resolvable via the Trusted Issuer List (see Section 8). This provides Relying Parties with cryptographic assurance that the EAA was issued by a registered QTSP.

## 2.5 Batch Issuance
In Phase 4, the Digital Room Key EAA and the Guest Summary EAA are issued in a single transaction. IDnow's OIDC4VCI endpoint supports batch credential issuance, allowing the hotel system to request both EAAs with a single pre-authorised code. The EUDIW receives and stores both credentials atomically.

## 2.6 Credential Schemas
The following preliminary attribute schemas define the data content of each EAA. These schemas will be formalised in a dedicated attestation rulebook (see Section 7.4).

## Booking Confirmation EAA

| Attribute | Type | Mandatory | Description |
|-----------|------|-----------|-------------|
| `booking_id` | string | Yes | Unique booking identifier from the Booking Platform |
| `hotel_name` | string | Yes | Name of the hotel |
| `hotel_id` | string | Yes | Unique hotel identifier |
| `guest_family_name` | string | Yes | Guest family name (from PID) |
| `guest_given_name` | string | Yes | Guest given name (from PID) |
| `check_in_date` | date | Yes | Expected check-in date |
| `check_out_date` | date | Yes | Expected checkout date |
| `room_type` | string | No | Room category (e.g., single, double, suite) |
| `number_of_guests` | integer | No | Number of guests on the booking |
| `iss` | string | Yes | Issuer identifier (IDnow) |
| `iat` | datetime | Yes | Issuance timestamp |
| `exp` | datetime | Yes | Expiry timestamp |
| `cnf` | object | Yes | Confirmation claim (wallet key binding) |
